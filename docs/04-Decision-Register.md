## Detail

### DR-001 — Storefront platform: custom web app
**Decision.** Build a custom storefront web app instead of Shopify Plus. Supersedes **SAD ADR-1** (Shopify Plus).
**Why.** Owner: Shopify Plus subscription too expensive.
**Implications.** We now build what Shopify provided: cart, checkout, payments (see DR-003), B2B primitives (DR-007), customer accounts, hosting. The **Azure integration layer is unaffected** — it was deliberately storefront-agnostic (SAD P2; the middleware OpenAPI is a plain HTTP contract any client calls), so T1–T5 carry over. Net change concentrates in the `storefront/` tier and the catalog/fulfilment **target** (DR-005).
**Alternatives considered.** Mid-tier commerce engine (non-Plus Shopify / BigCommerce / headless commercetools / Medusa.js) — keeps buying the PCI-hard checkout while swapping only the storefront tier; recommended as the fallback if the custom build's cost (esp. PCI) exceeds the Plus saving. Left **Open** as a pivot option since the middleware doesn't care which storefront calls it.

### DR-002 — Frontend + BFF stack
**Decision.** React + TypeScript SPA (Vite) talking to an ASP.NET Core **backend-for-frontend (BFF)** on .NET 10; the BFF calls the middleware via APIM using the T4 resilient clients.
**Why.** React has the richest B2B-commerce component ecosystem; a BFF keeps tokens/secrets server-side (DR-004), aggregates middleware calls, terminates auth, and avoids exposing APIM/CORS to the browser. BFF in .NET matches the chosen stack/team.
**Alternatives.** Blazor (all-.NET, weaker commerce ecosystem); Next.js SSR (heavier hosting; SEO is low-value for an auth-gated B2B catalog, so a SPA suffices).

### DR-003 — Payments & PCI
**Decision.** Use a hosted payment integration that keeps us in **PCI DSS SAQ-A** (card data entered into the provider's hosted fields/redirect; never touches our servers, Azure, or FinOps). Default **Stripe** (Payment Element / hosted Checkout), behind an `IPaymentProvider` abstraction so Adyen/Braintree/etc. swap cleanly.
**Why.** PCI is the dominant cost/risk of going custom (SAD §9/§10 previously contained it inside Shopify). SAQ-A via hosted fields minimizes scope.
**Owner-confirmed (2026-06-26): Stripe.** Build proceeds against `IPaymentProvider` with Stripe (Payment Element / hosted Checkout, SAQ-A) as the implementation. Remaining: commercial terms + confirm SAQ-A (not SAQ-D) at implementation time (S5).

### DR-004 — Customer identity / auth
**Decision.** Microsoft Entra SSO; OIDC auth-code flow terminated at the BFF as a confidential client; tokens held server-side in the BFF session, never in the browser.
**Why.** Already an Entra/Azure shop; aligns with SAD §9 (Entra, least privilege).
**Owner-confirmed (2026-06-26): Azure Entra SSO.** Sub-detail to settle at implementation time (S1/S6): tenant model for external B2B customers — **Entra External ID** (CIAM) for customers who aren't in the org tenant, vs Entra ID B2B guests. One-line difference at the BFF's OIDC config; defaulting to Entra External ID for external customers unless told otherwise.

### DR-005 — Storefront catalog store
**Decision.** The custom app owns its own catalog **read-store**, fed by the existing BYOD→storefront catalog sync (T5). For Phase 1 the sync target (`mocks/shopify-sim` + `IShopifyCatalogSink`) **stands in** for that store; it is renamed to storefront-neutral names when the real catalog store/API is built (avoids churn on green T5 code now).
**Why.** Read/write separation (SAD P2) still holds — browse comes from a replica the storefront owns, never FinOps/OData.

### DR-006 — Hosting
**Decision.** SPA on Azure Static Web Apps (+ CDN); BFF on Azure App Service or Container Apps; both behind APIM/Front Door. Added to Bicep in the storefront-infra task (S7).

### DR-007 — Phase-1 B2B scope

### DR-009 — Reservation atomicity at the checkout gate
**Decision.** `/cart/reserve` reserves the whole cart **all-or-nothing** (T6). If any line cannot be fully reserved, any partial reservations already placed are released, and the response returns each line's reservation/shortfall so the storefront can offer reduce / backorder / split, then re-reserve.
**Why.** Prevents reservation leak / holding stock for a cart that cannot be committed; keeps the soft-reservation lifecycle clean (TDD §7.1). The HTTP surface returns 409 with the per-line breakdown on shortfall.
**Alternative.** Hold partial reservations and let the caller top up — rejected: risks orphaned/leaked reservations and complicates release.

### DR-012 — T11 scenario coverage
**Decision.** Close out the T11 scenario matrix against the Phase-1 mocks with the smallest honest production changes:
- **Catalog-attribute join.** `CartLineInput` now carries `itemClass`, `backorderable`, `madeToOrder`, `discontinued` (advisory hints from the T5 catalog); `CartAvailabilityService` feeds them to the band calculator so **backorder / made-to-order / discontinued** band through the live stack (previously the service stubbed these to `false`). The live ATP read stays authoritative (Golden Rule #5).
- **Returns / cancellation.** The status-event contract gains `returned` and `cancelled`; `StorefrontOrderStatus` gains `Returned`/`Cancelled`; `StatusSyncService` maps them (fulfilment history preserved on return).
- **Price integrity (TDD §9).** `PriceIntegrityPolicy` compares each line's locked price to the current FinOps price within a **config-driven** tolerance (`PriceIntegrity:ToleranceFraction`); out-of-tolerance lines are routed to CSR review as a permanent failure (reservation released via the existing saga). Disabled unless the section is configured, so the locked price remains authoritative by default.
- **Kits / advance-block / multi-warehouse** need no new production code: a kit is modeled as its exploded component lines reserved all-or-nothing (DR-009); advance/block uses the IVS allocation endpoint; multi-warehouse uses per-(site, location) ATP that already exists in the IVS interface.

**Coverage.** `tests/integration/T11ScenarioTests.cs` (13 scenarios) + `tests/unit/PriceIntegrityPolicyTests.cs`. Min-qty/order-multiple/UoM remain proven at the BFF (`BffCartCheckoutTests`); the reservation state machine by `AvailabilityTests`/`ReservationReleaseTests`.
**Status (2026-06-27): T11 delivered.** Full suite 158 green (94 unit + 64 integration).

### DR-014 — Sales-order number ownership (Open Decision #3)
**Decision.** D365 FinOps owns and generates the sales-order number (its number sequence on `SalesOrderHeadersV2`); the portal never generates or predicts it. Because writeback is async (Golden Rule #7), the portal mints a durable order id at placement (the GUID that is already `OrderRequest.IdempotencyKey` / `OrderStatusResponse.OrderId`) and shows that immediately; the FinOps SO number is back-filled as a secondary "ERP reference" once writeback returns it (`OrderStatusResponse.SalesOrderNumber` is nullable until then). Never render a fabricated/"pending" SO number.
**Why.** Only FinOps can guarantee uniqueness/continuity/legal formatting; a shadow sequence would collide/drift and break the system-of-record boundary. The portal id doubles as the writeback idempotency key, so it must exist before the ERP issues a number. The contracts already encode this two-identifier model — this ratifies it.
**Status.** Largely already true in code. The customer-facing primary reference is the portal order id; a small follow-up surfaces both ids on order detail (config flags `OrderReference:*` recorded for that). Validate the real sequence format/sync-vs-async return against the Phase-2 sandbox.

### DR-015 — Reservation TTL + release triggers (Open Decision #4)
**Decision.** `Ivs:ReservationTtlSeconds = 900` (15 min) — covers checkout-gate dwell (PO entry + payment redirect) while freeing AFR on abandonment within the quarter-hour. Sweep cadence every 2 min (`ReservationReleaseFunction`; ~17 min worst-case over-hold), overridable at deploy via `Ivs:ReservationSweepCron`. **Release triggers** (all idempotent, Soft→Converted/Released, TDD §7.1): order placed → Converted; TTL expiry / abandoned cart → Released by the sweep; explicit cancel / checkout back-out → `/cart/release`; permanent writeback failure (credit reject / DLQ-exhausted) → Released; all-or-nothing partial-reserve rollback → Released (DR-009); reserved-line reduce/remove → release the superseded reservation before re-reserving.
**Why.** Reservations are placed only at the gate (post-intent), IVS soft reservations don't auto-expire (the integration layer owns expiry — hence the registry + sweep). **Depends on DR-011's durable registry** to be meaningful across the (separate) release Function app. Re-tune 900s from real checkout-dwell telemetry in Phase 2.

### DR-016 — Per-class buffers + LowStock threshold (Open Decision #5)
**Decision.** ABC-velocity buffers on the `Availability` config: `ClassBuffers A=8, B=4, C=1` units, `DefaultBuffer=4` (unknown/empty class treated as B, **not 0**), `LowStockThreshold=10` (published ATP `>10` InStock, `1–10` LowStock, `≤0` Backorder-if-backorderable else Unavailable). Buffers are **flat unit counts**, not percentages (percentages round to zero on the thin-stock low-count parts where the last-unit race matters most).
**Why.** The buffer absorbs the window between the advisory synced/published ATP and the authoritative live check; oversell risk scales with demand velocity, so fast movers (A) get the largest buffer. Values are advisory display-only (Golden Rules #2/#4) — too-small degrades to a benign gate downgrade, not oversell. **Tuning loop:** emit per-gate-check telemetry `{itemClass, site, publishedAtp, liveAtp, band, outcome}`; target 1–3% gate-downgrade per class, +50% on breach, −1 step when well under 1%. Tune `LowStockThreshold` separately against LowStock fulfilment rate.
**Applied.** `itemClass` is not yet a catalog field (defaults to B via `DefaultBuffer`); `backorderable`/`discontinued` now flow from the catalog through the BFF validate path so bands are correct. Dev seed ATPs adjusted for band variety under the buffers.

### DR-017 — Multi-warehouse availability (Open Decision #7)
**Decision.** **Branch-specific** by default: bands, the live gate, the soft reservation, and the FinOps order line are all evaluated against a single concrete `(site, location)` per line. Aggregate (summed-across-sites) numbers are **never** used to gate checkout or place a reservation — at most an advisory "available at other branches" browse hint. Site is resolved server-side by precedence: explicit line site → customer default (home branch) → tenant fallback, stamped on the line **before** the ATP read so validate→reserve→order use the identical tuple. **No cross-site split** in Phase 1 (a line ships whole from one branch or is short).
**Why.** IVS keys ATP/AFR and every reserve by `(productId, site, location)` — there is no aggregate-reserve primitive, so an aggregate band could promise stock no single branch can honor. Keeps DR-009 all-or-nothing clean and matches B2B trade-counter expectations (no surprise split shipments). The code already evaluates per `line.Site`; Phase 2 adds the `Ivs:LocationBySite` map + `Ivs:SiteResolution:*` precedence (recorded) and tunes to real sandbox dimensions (config only).

### DR-018 — Customer-facing representation of terminal order statuses
**Decision.** Terminal negative fulfilment events get **dedicated** customer-facing statuses, not the generic `WrittenBack` bucket. The middleware `OrderStatus` contract enum gains `Cancelled` and `Returned`; `OrderStatusMapper.MapStatus` maps `StorefrontOrderStatus.Cancelled → Cancelled` and `Returned → Returned`. `Packed`/`Updated` keep mapping to `WrittenBack` (shown as "Processing") — they are genuinely post-writeback/pre-ship, so "Processing" is honest. The SPA renders `Cancelled` with a danger tone and `Returned` with a warn tone. The status `message` is now a **sentence-cased customer phrase** (e.g. "Partially shipped · 3 on backorder"), never the raw `StorefrontOrderStatus` enum name.
**Why.** Surfaced via Golden Rule #12 (don't silently guess customer-facing behavior). The order-status sync pipeline can emit `Cancelled`/`Returned` (from `cancelled`/`returned` FinOps fulfilment events), but the mapper previously collapsed every non-shipped status into `WrittenBack`, so a **cancelled or returned order showed the customer a reassuring "Processing" badge** — a misrepresentation, compounded by the detail page printing the raw enum name beneath it. Distinct terminal statuses are the faithful representation; tone/label and the phrase map keep the UI honest. Phase 2 may add richer terminal copy (return tracking, RMA links) once FinOps return semantics are validated.

### DR-019 — Pay-on-account (net terms) settlement + PO number
**Decision.** B2B customers with **approved credit** can settle an order **on account** (net terms) instead of being forced through a card. A `paymentMethod` ("Card" | "OnAccount", string passthrough so it crosses the order/message/ERP contracts unchanged) is added to `OrderRequest` → `OrderInboundMessage` → FinOps `SalesOrderHeaderRequest`, plus an optional header-level `purchaseOrderNumber` (≤50 chars). The BFF `PayRequest` gains `PaymentMethod` (BFF enum `SettlementMethod`, default `Card`) + `PoNumber`. **On-account is gated server-side on the re-resolved credit decision being `Approved`** — `RequiresApproval`/`Blocked` are refused (`PayResult("CreditDeclined")`), the client's choice is never trusted. The card branch is skipped entirely for on-account (no token, no charge). `CheckoutResult.AllowOnAccount` (= decision is Approved) tells the SPA when to offer the option; the SPA shows a tender selector + optional PO input. PO is **optional** in Phase 1 (no per-customer "PO required" policy exists yet). `ODataOrderClient.CreateHeaderAsync` now posts the full header (customer, currency, paymentMethod, PO, correlation, idempotency) — previously it dropped everything but `customerAccount`.
**Why.** "Everyone is forced through a card; the credit decision gates nothing on payment" was a go-live blocker for a B2B portal — net terms is the norm for trade accounts. Gating on `Approved` only (fail-closed) mirrors the existing checkout credit gate and is safe without a numeric credit limit (none exists in the Phase-1 contract/mock; `availableCredit` is a forward Phase-2 optional). **Phase-2 follow-ups:** surface `availableCredit` and enforce an on-account cap; per-customer "PO required" policy; an approval/draft-order path for `RequiresApproval`.

### DR-020 — Order + charge idempotency key
**Decision.** The order idempotency key is **derived deterministically** from `SHA-256(customerAccount | sorted reservationIds)` instead of a fresh `Guid` per `/checkout/pay` call. The same key is passed to the payment provider (`PaymentRequest.IdempotencyKey` → Stripe `RequestOptions.IdempotencyKey`) and to the order writeback (`OrderRequest.IdempotencyKey`, which the writeback already de-dups on). A retry/double-click with the same reserved cart therefore reuses one key — Stripe returns the original charge and writeback returns the original order, instead of double-charging and duplicating.
**Why.** `BuildOrder` minted `Guid.NewGuid()` on every call, so a retried pay created a **new** order; and `StripePaymentProvider` created the PaymentIntent with **no** idempotency key, so a concurrent double-submit could **double-charge**. Deriving from `(customer, reservations)` ties the key to the exact thing being paid for, so honest retries collapse and distinct orders stay distinct (order-insensitive over the reservation set, customer-scoped).
**Review hardening (adversarial pass).** The reservation set is now held **server-side**: `/checkout/start` records it in an `ICheckoutSessionStore` (in-memory Phase 1, like the cart/registry — DR-011) and `PayAsync` reads it from there, **ignoring** any client-supplied `reservationIds`. Paying without a live gate session returns `NoReservation` (re-run the gate). This (a) restores reserve-before-commit (Golden Rule #5) — an order can't be submitted with no/forged reservations; (b) makes the idempotency key un-forgeable (no client-controlled empty-list → random-key bypass); (c) removes an NRE on omitted `reservationIds`. The **currency is pinned server-side** from `Bff:Currency` (default GBP) for both the charge and the locked price (the client currency is ignored, matching the amount). `StripePaymentProvider` catches the concurrent `idempotency_key_in_use` 409 and returns a clean "still processing" result instead of a 500. The SPA no longer strands a failed pay on the Confirmation step — `CreditDeclined`/`PaymentFailed` stay on the payment step (switch tender + retry), `NoReservation` returns to Review.

### DR-021 — VAT/GST: FinOps owns tax; the portal surfaces it
**Decision (owner-directed).** The portal does **not compute** tax. Tax/VAT/GST is **owned by D365 FinOps** and flows to the portal through the pricing/credit `resolve` response: `ResolvedLine` gains optional `taxRate` + `taxAmount`, carried into `PricedLine` (with a computed `GrossEffectivePrice = net + tax`). The checkout charges the **gross** (`PaymentService` sums `GrossEffectivePrice`); the order's locked unit price stays **net**, so writeback price-integrity still compares like-for-like. The SPA shows **Subtotal / VAT / Total** at the checkout gate and labels it "Tax confirmed by Dynamics 365". The order writeback carries **no portal-computed tax** (FinOps recomputes on its own sales order). Phase 1: `pricing-credit-sim` stands in for FinOps tax via a config rate `Tax:Rate` (default 0.20); absent tax → 0 (gross == net), so the portal never invents tax.
**Why.** Owner chose "FinOps owns tax" over portal-computed: tax engines (jurisdiction, reverse-charge, exemptions, registration) are a FinOps/ERP responsibility, and computing it in the portal would risk a mismatch with the authoritative sales-order tax. Surfacing FinOps tax keeps a correct customer-facing gross total without duplicating tax logic. **Phase 2:** wire the real FinOps tax into the `resolve` response (jurisdiction from ship-to); reverse-charge / zero-rate then arrive as `taxAmount: 0` with a FinOps-provided treatment, no portal change.

### DR-022 — BFF security/ops hardening
**Decision.** The BFF gains (prod-readiness #6): **rate limiting** (a global per-client fixed-window + a stricter "sensitive" policy on `/api/auth/login|logout` and `/api/checkout/pay`; config `RateLimit:PermitPerMinute`/`SensitivePerMinute`), **security headers** incl. a Stripe/Google-Fonts-aware **CSP** (config `Security:ContentSecurityPolicy`), **HSTS** outside Development, **structured request logging** (`UseHttpLogging` — method/path/status/duration only; never bodies, the auth cookie, or `X-Dev-Customer`), **forwarded-headers** handling (trusts only `ForwardedHeaders:KnownNetworks`, so the rate limiter sees the real client IP behind APIM without X-Forwarded-For spoofing), and **cookie hardening** (HttpOnly + Secure + SameSite=Lax). The **SPA document** CSP lives in `staticwebapp.config.json` (the SPA is served by Static Web Apps, not the BFF — the BFF CSP is defence-in-depth for `/api`).
**Why.** The BFF previously emitted zero logs and had no abuse controls; the checkout-bearing SPA shipped with no CSP. Surfaced via an adversarial review that caught three real issues in the first cut: the CSP only reached `/api` (not the SPA HTML) → added to the SWA config; the rate limiter ran *before* authentication so the per-customer partition key was dead (keyed by IP only) → moved `UseRateLimiter` after `UseAuthentication` (verified by a per-customer-bucket test); and no forwarded-headers handling → all traffic collapsed to one IP partition behind a proxy. **Remaining for deploy:** App Insights/OpenTelemetry exporter (the structured logs are emitted; wiring an AI connection string is a deploy-time, live-Azure step); per-proxy `KnownNetworks` values.

### DR-023 — Numeric credit limit/remaining + profile (account/billing depth)
**Decision.** The pricing/credit `resolve` response now carries FinOps-owned `creditLimit` + `availableCredit` (Money) — surfaced unchanged by the portal (it never computes credit, same posture as tax — DR-021). They flow `ResolveResponse → PricingResult → CartPricingResult → CreditStanding`; the Account page shows the **credit limit + remaining headroom** with a usage meter, plus a **read-only profile** (name/email/account from the session; editing delegated to Entra). On-account checkout is now **capped by headroom**: `CheckoutResult.AllowOnAccount` requires `availableCredit ≥ gross order value`, and `PaymentService` re-checks it authoritatively at pay time (`CreditDeclined` when an order exceeds remaining credit) — closing the DR-019 Phase-2 follow-up. Phase-1 `pricing-credit-sim` emits the numbers (config `Credit:LimitDefault`/`AvailableDefault`, seedable per customer); absent → unknown, no numeric cap enforced (fails open to the coarse decision only).
**Why.** "Numeric credit limit/remaining (not even in the contract today)" was a high B2B gap, and without it on-account had no value cap (DR-019 noted this). Surfacing FinOps numbers (not computing them) keeps the ERP authoritative. Profile is read-only because identity is owned by Entra (DR-004). **Remaining #5:** shipping/billing address book (a separate CRUD feature).

### DR-024 — Transactional email
**Decision.** An `IEmailSender` abstraction (config `Email:Provider`, mirroring the `IPaymentProvider` pattern, DR-003) with a Phase-1 `LoggingEmailSender` that logs the message instead of delivering it (no SMTP/provider needed locally). The BFF sends an **order-confirmation** email on `OrderPlaced`, to the customer's email from the session claim (never the client), with the order reference — fired from `PaymentService` and **swallowing any send failure** (a placed order must never fail because email did). Plain-text bodies, minimal PII (recipient + order ref).
**Why.** "Transactional email — order confirmation, …" was a high gap. The abstraction lets a real provider (SendGrid/SMTP/Azure Communication Services) drop in at deploy with no code change. **Remaining:** shipment/tracking + invoice emails (the shipment trigger lives in the Sync app's status pipeline, which needs a customer-email lookup); a real provider + sender identity at deploy.

### DR-025 — Cart must match the reservation at pay time
**Decision.** `/checkout/start` now snapshots the **exact cart lines** it reserved into the `CheckoutSession` (alongside the reservation ids). `PaymentService.PayAsync` rejects with **`CartChanged`** (re-run the gate) if the live cart no longer equals that snapshot, and builds the order + pricing from the **reserved snapshot** (aligned with the reservation order) rather than the live cart. The SPA treats `CartChanged` like `NoReservation` — back to Review.
**Why.** Surfaced by the DR-023 review: `PayAsync` built the order from the *current* cart but reused the gate's reservation set, and `OrderInboundMessageMapper` maps reservation references to lines **positionally** — so if the cart changed between gate and pay (e.g. a second tab added a line), the extra line reached writeback with an **empty** `reservationReference`, bypassing reserve-before-commit (Golden Rule #5). Snapshotting + comparing makes the order provably equal to what was reserved and priced. Pre-existing (not introduced by the pay-on-account/VAT work); fixed here.
